## Deploy And Contribution Workflow

The repo is private. Production deploys run from GitHub Actions using the Vercel CLI, not Vercel's native Git integration.

Current production path:

1. A commit lands on `main`.
2. `.github/workflows/deploy.yml` runs.
3. The workflow installs dependencies, runs `vercel pull`, `vercel build --prod`, and `vercel deploy --prebuilt --prod` using `VERCEL_TOKEN`, `VERCEL_ORG_ID`, and `VERCEL_PROJECT_ID` from GitHub Secrets.
4. Vercel aliases the deployment to https://www.bobsbigmeats.com/.

Rob's supported workflow is **PR-based, not direct-push**:

1. Rob creates a branch.
2. Rob opens a pull request.
3. Evan merges the PR into `main` with a **merge commit**.
4. The Evan-authored merge commit triggers the GitHub Action and deploys.

**Merge commits only. Never squash, never rebase.** Squash and rebase merges can change commit authorship in ways that re-trigger Vercel Hobby's per-commit-author restriction for private repos. Rob pushing directly to `main` may stall or fail in Vercel; the tested path is branch -> PR -> Evan merge commit -> deploy.

Rob does not currently have Vercel or Cloudflare access. If a bad merge reaches production, Evan should revert the merge commit and push the revert to `main`; the revert deploys through the same GitHub Action path. Do not tell Rob to recover by pushing directly to `main`.

Future exit: once Rob has his own Vercel account connected to this repo or a fork, the PR-as-gatekeeper constraint can go away and Rob can use a simpler direct deploy workflow.

Token rotation note for Evan: `VERCEL_TOKEN` lives only in GitHub Secrets. If it leaks or expires, regenerate it in Vercel, update the repository secret, and rerun the deploy workflow.

First Edit

This is a low-stakes walkthrough: change one visible menu tag, open a PR, and let Evan merge it. You will see the full loop without touching anything fragile.

The workflow is PR-based. Do not push directly to main. Evan is the merge gatekeeper for now because Vercel Hobby rejects or stalls on some non-owner private-repo commits.
  1. Create a branch: git checkout -b rob-menu-tag-test.
  2. Open app/menu/page.tsx.
  3. Find the menu-brisket-tag slot from CONTENT.md: it currently says tag: "King".
  4. Change it to another short label, for example tag: "Crown".
  5. Run npm run dev and check http://localhost:3000/menu.
  6. Commit it: git add app/menu/page.tsx && git commit -m "test: update brisket menu tag".
  7. Push the branch: git push -u origin rob-menu-tag-test.
  8. Open the branch on GitHub and click Compare & pull request. Put Evan as reviewer.
  9. Evan reviews and merges with Create a merge commit. Not squash. Not rebase.
  10. GitHub Actions deploys automatically. The live site usually updates shortly after the merge.
