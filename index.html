## How To Work On This Site

**The one rule that matters: iterate locally until the site is exactly right, then ship the whole batch in a single pull request. Never open a pull request for an unfinished change. Never open one pull request per tweak.**

### The working loop

1. **Clone and run it locally.** Clone the repo, run `npm install`, then `npm run dev`. The site is now running at `http://localhost:3000`. This is the playground.
2. **Iterate freely on localhost.** Edit a file, save, and the local site updates instantly. Change copy, swap placeholder content, adjust layout — as many times as wanted. None of this touches the live site and nobody is reviewing it. This is where all the real work happens.
3. **Get it exactly right.** Keep iterating on `http://localhost:3000` until the site looks exactly how Rob wants it. There is no cost to experimenting and no rush — it is all local and private.
4. **Pre-flight check: `npm run build`.** Before shipping, run `npm run build`. If it passes, the work is deploy-safe. If it fails, fix it locally first — a change that works under `npm run dev` can still break a production build, and the pull request should arrive deploy-ready.
5. **Ship the whole batch in ONE pull request.** When the work is genuinely finished, create a branch, commit everything, push the branch, and open a single pull request containing all of it. One pull request per "this is ready" moment — never one per change.
6. **Evan merges.** Evan reviews the pull request and merges it with a merge commit. That triggers the deploy; the change is live at https://www.bobsbigmeats.com within a minute or two.

### Why it works this way

- `npm run dev` gives instant visual feedback. A pull request gives none — it is invisible until Evan merges it. Every "does this look right?" decision belongs on `localhost`, never in a stream of pull requests.
- One batched pull request means Evan reviews once, merges once, and deploys once — instead of being pulled in for every micro-edit.
- It keeps the deploy path clean: fewer merge commits, fewer deploys, and less exposure to the Vercel commit-author issue described in Deploy Mechanics.

**Hard check before opening any pull request: has this exact change been seen working on `http://localhost:3000`? If not, stop and run it locally first.**

The repo ZIP download is fine for an agent to *read* the code, but real work needs a proper `git clone` — a branch and a pull request cannot come from a loose ZIP.

## Deploy Mechanics

The repo is private. Production deploys run from GitHub Actions using the Vercel CLI, not Vercel's native Git integration.

How a deploy happens:

1. A commit lands on `main` — in practice, a pull request merged by Evan (see How To Work On This Site).
2. `.github/workflows/deploy.yml` runs.
3. The workflow installs dependencies, runs `vercel pull`, `vercel build --prod`, and `vercel deploy --prebuilt --prod` using `VERCEL_TOKEN`, `VERCEL_ORG_ID`, and `VERCEL_PROJECT_ID` from GitHub Secrets.
4. Vercel aliases the deployment to https://www.bobsbigmeats.com/.

**Merge commits only. Never squash, never rebase.** Squash and rebase change commit authorship in ways that re-trigger Vercel Hobby's per-commit-author restriction for private repos. Evan merges every pull request with a merge commit so the commit on `main` is Evan-authored. Direct pushes to `main` may stall or fail in Vercel; the tested path is branch, then PR, then Evan merge commit, then deploy.

Rob does not currently have Vercel or Cloudflare access. If a bad merge reaches production, Evan reverts the merge commit and pushes the revert to `main`; the revert deploys through the same GitHub Action path. Do not tell Rob to recover by pushing directly to `main`.

Future exit: once Rob has his own Vercel account connected to this repo or a fork, the PR-as-gatekeeper constraint can go away and Rob can use a simpler direct deploy workflow.

Token rotation: `VERCEL_TOKEN` lives only in GitHub Secrets. If it leaks or expires, regenerate it in Vercel, update the repository secret, and rerun the deploy workflow.
