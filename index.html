## Deploy And Contribution Workflow

The repo is private. Production deploys run from GitHub Actions using the Vercel CLI, not Vercel's native Git integration.

Current production path:

1. A commit lands on `main`.
2. `.github/workflows/deploy.yml` runs.
3. The workflow installs dependencies, runs `vercel pull`, `vercel build --prod`, and `vercel deploy --prebuilt --prod` using `VERCEL_TOKEN`, `VERCEL_ORG_ID`, and `VERCEL_PROJECT_ID` from GitHub Secrets.
4. Vercel aliases the deployment to https://www.bobsbigmeats.com/.

Supported contribution workflow:

1. Create a branch.
2. Open a pull request.
3. Evan merges the PR into `main` with a **merge commit**.
4. The Evan-authored merge commit triggers the GitHub Action and deploys.

**Merge commits only. Never squash, never rebase.** Squash and rebase merges can change commit authorship in ways that re-trigger Vercel Hobby's per-commit-author restriction for private repos. Direct pushes to `main` may stall or fail in Vercel; the tested path is branch -> PR -> Evan merge commit -> deploy.

Rob does not currently have Vercel or Cloudflare access. If a bad merge reaches production, revert the merge commit and push the revert to `main`; the revert deploys through the same GitHub Action path. Do not tell Rob to recover by pushing directly to `main`.

Future exit: once Rob has his own Vercel account connected to this repo or a fork, the PR-as-gatekeeper constraint can go away and Rob can use a simpler direct deploy workflow.

Token rotation: `VERCEL_TOKEN` lives only in GitHub Secrets. If it leaks or expires, regenerate it in Vercel, update the repository secret, and rerun the deploy workflow.
